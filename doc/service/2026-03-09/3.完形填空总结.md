我捋一下思路，就是前端创建完形填空的时候，是由一个父题（负责渲染材料）和多个子题构成，然后子题由parent Id和clozeIndex内部排序，然后后端先插入父题，拿到id，作为子题的父题后续关联出现，然后查询题目列表的时候只出现父题，如果插入试卷时if判断发现题型是cloze时，内部找到所有的子题，然后制作orderIndex，但父题的orderIndex为Null，同父题一起塞到数据库，渲染时正常渲染前端渲染的时候父题orderindex为空则只提取题干材料不做为数组内容，然后继续按下方的子题展示


一句话版最终流程（建议你写到总结里）
题库创建：先新增父题 → 拿到父题 id → 批量新增子题（带 parentId 和 clozeIndex）。
组卷保存：试卷内容里保存父题 id + 子题 id（父子都在试卷结构中）。
下发试卷/回顾试卷：后端根据试卷内容查出所有题 → 转 DTO 时重算 itemOrder（父题 null，子题/普通题连续）→ 前端渲染父题材料块，并在其下用 parentId 找子题按 clozeIndex 排序渲染；左侧题号/可作答列表只用“非父题”的题目。
你这段总结里如果把“同父题一起塞到数据库”改成“试卷内容结构保存父题+子题的题目ID”，以及把“靠 itemOrder null 判断父题”改成“靠 questionType===6 判断父题”，就完全严丝合缝了。
